📌 Level 04 · Modelling · L4.01 — Model Architecture & Best Practice

A financial model is only as trustworthy as its architecture. Before you build a single formula, the decisions you make about structure, conventions, and organisation will determine whether your model survives six months of updates — or becomes unauditable within two weeks.

The Tab Architecture

Separate inputs from calculations from outputs. This is the single most important structural rule in financial modelling. A model where assumptions are scattered across multiple tabs is impossible to audit, update, or scenario-test reliably.

Colour Conventions

Standard IB model colour convention: BLUE TEXT: Hardcoded inputs (assumptions you typed manually) BLACK TEXT: Formula-driven cells (calculations) GREEN TEXT: Links to other tabs or external sources RED TEXT: Error flags or broken links GREY TEXT: Supporting notes or documentation Background fills (some firms): LIGHT BLUE: Assumption / input cells WHITE: Calculation cells LIGHT GREY: Non-editable reference cells

These conventions must be applied consistently throughout the model. An auditor opening your model should be able to tell in three seconds whether any cell is an input or a formula.

The Formula Discipline Rules

Rule 1 — Never hardcode in forecast columns. Every cell in a forecast year must be a formula. The only exception: the assumption tab itself. If you find yourself typing "220" into a Year 3 revenue cell, stop — that number should be "=prior_year × (1 + growth_rate_assumption)".

Rule 2 — One row, one concept. Each line of the model does one thing. Don't bundle two calculations into one row. Debt interest for Tranche A and Tranche B should be two separate rows, summed to a subtotal.

Rule 3 — Consistent sign convention. Decide upfront: is capex positive or negative in the CFS? Is interest expense positive or negative on the IS? Apply your convention everywhere without exception. Mixed sign conventions are responsible for more model errors than any other single issue.

Rule 4 — Reference, don't retype. If a number from one tab is needed in another, use a cell reference (=IS!B12) — never retype the value. Retyping creates two versions of the same number that will inevitably diverge.

Version Control — The Rule That Saves Careers

Every time you send a model externally, save a new version file: "CompanyName_Model_v2.1_20250420.xlsx". Never overwrite the last sent version. When a client says "can you send me the version from last Tuesday?" — and they will — you need to find it in 30 seconds. One analyst who can immediately produce the version a client is referencing earns enormous credibility. The one who cannot is in a very uncomfortable position.

The Structural Mistakes MDs Can Spot Immediately

No balance sheet check. If there is no CHECK row at the bottom of the balance sheet, the model has not been verified to balance.

Assumptions scattered across tabs. Changing a margin assumption requires hunting through five different tabs — the model cannot be scenario-tested reliably.

Formulas that skip rows. =B14+B16 (skipping B15) is a lurking error — if B15 should be included in a sum but was accidentally missed, the model gives wrong totals for years without anyone noticing.

In a Real Bank

Most bulge-bracket banks have proprietary model templates — pre-built tab structures, pre-set colour conventions, and standard formula layouts. When you join, your first task is usually to learn the house template before building anything from scratch. The template exists because standardised models are faster to review (MDs know exactly where to look) and harder to break (the balance checks are already built in). Follow the house template exactly — do not "improve" it without senior approval.

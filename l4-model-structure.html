📌 Level 04 · Modelling · L4.01 — Model Architecture & Best Practices

A financial model is only as good as its architecture. The structure determines whether the model is reliable under pressure, legible to colleagues, and defensible to clients. These conventions are not optional in professional IB — they are the baseline.

The Golden Rules of Model Architecture

RuleWhat it meansWhy it matters
Blue for inputsAll hard-coded assumptions are coloured blue. No exceptions.Anyone opening the model immediately knows which cells drive the outputs and which to change when assumptions update.
Black for formulasCalculated cells are black. They reference inputs — never contain hard-coded numbers except 1, 0, 100, or 365.Prevents assumptions being buried in formulas, invisible to auditors.
One direction of flowData flows left to right (earlier years to later years) and top to bottom. No cell references a cell to its right or below it in the same column.Prevents invisible circular references. Makes the model readable in one pass.
Separate tabs by functionAssumptions, P&L, Balance Sheet, CFS, Debt Schedule, Working Capital, Capex, Summary/Output are all separate tabs.No tab becomes unwieldy. Every component can be audited independently.
Consistent column structureHistorical years on the left (grey background), forecast years to the right (white/no fill). A clear visual break between them.Prevents mixing actual data into forecast formulas — a major source of errors.
Balance check on every tabA visible row or cell at the bottom of every connected tab confirming outputs are consistent.Errors surface immediately where they occur, not three tabs downstream.

Tab Structure — the standard IB layout

Standard model tab order: [Cover] — title, date, version, confidentiality [Inputs] — all assumptions in one place (growth rates, margins, capex %, tax rate) [P&L] — income statement, historical + forecast [BS] — balance sheet, historical + forecast [CFS] — cash flow statement, historical + forecast [Debt] — debt schedule, amortisation, interest, cash sweeps [WC] — working capital: AR, inventory, AP days drivers [Capex] — capex schedule, PP&E roll, depreciation waterfall [Val] — DCF, comps summary, football field [Output] — management summary: key metrics, leverage, cover, IRR [Scenarios]— base/downside/severe toggles, sensitivity tables

Version Control and Documentation

Every model sent outside the team must have a version number and date in the filename. Example: CompanyX_LBO_Model_v4_2024-11-15.xlsx. Never overwrite a previously sent version — always save as a new file. If the client has v3 and you send v4, they must be able to compare the two.

The Cover tab should include: model purpose, date, version, author, key assumptions summary, and a disclaimer. In a live deal, the Cover tab is what an MD reads first when a model arrives at 11pm — make it informative.

The Audit Test

Before sending any model: open it fresh, close all tabs except the Output tab, and try to trace every key output back to a source assumption without opening the Inputs tab. If you cannot do this within 5 minutes, the model is not auditable. Reorganise until it is. An MD who cannot find the revenue growth assumption during a client call will not be happy — and will remember.

The Most Common Architecture Mistakes

Linking across workbooks. External links to other Excel files break when files move or are renamed. Always embed all data within a single workbook. If you must reference external data, hard-code it into a clearly labelled "Source Data" tab with the source noted alongside.

Merged cells. Merged cells break array formulas, cause copy-paste errors, and make data tables fail. Never merge cells in a live model — use formatting (borders, bold) to achieve visual grouping instead.

Hiding rows or columns. Hidden rows containing intermediate calculations or assumptions become invisible in a client meeting and are the first place errors hide. Use outline grouping (Group → Rows) instead — it shows that rows are grouped, not hidden.

In a Real Bank

Most bulge-bracket banks maintain proprietary model templates for each product type — M&A, LBO, IPO, DCF. First-year analysts spend their first 2–3 weeks learning the bank's specific template before touching a live deal. The template enforces all the conventions above. When you eventually build from scratch (for a boutique or for a special-purpose model), you build on top of the template architecture — not from a blank workbook.
